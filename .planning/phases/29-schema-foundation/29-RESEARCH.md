## Summary

Phase 29 delivers two non-destructive schema changes that all subsequent v2.2 phases depend on: a nullable `archived_at TIMESTAMPTZ` column on the events table, and a new `recurring_series` table with a `series_id` FK back on events. Both changes can be introduced with zero impact on existing queries — nullable columns with no default value leave all existing rows with NULL, and the new table has no rows until Phase 31 (detection) populates it.

The project uses Drizzle ORM 0.45.1 with drizzle-kit 0.31.9, Neon Postgres (serverless HTTP driver), and a `drizzle/` folder containing numbered migration SQL files generated by `npm run db:generate` and applied via `npm run db:migrate`. The migration workflow is fully established with 11 prior migrations as reference. TypeScript types flow automatically through `InferSelectModel` in `src/types/index.ts` — adding a column to the schema definition propagates the type to `Event` with no manual edits.

The critical observation from STATE.md is confirmed by code inspection: `event_date` is stored as plain `TIMESTAMP` (no timezone). This is pre-existing and not in Phase 29's scope to change, but it means the archival cron (Phase 30) must apply an explicit Atlantic offset in application code when computing the archival threshold.

**Primary recommendation:** Use `timestamp('archived_at', { withTimezone: true })` for the new column (TIMESTAMPTZ as required by ARCH-01), follow the established `ADD COLUMN` pattern from prior migrations, and define `recurring_series` as a standard `pgTable` with a `uniqueIndex` on `(venue_id, normalized_performer)`.

### Pattern 1: Adding a Nullable Column (Non-Destructive)
**What:** Add `archived_at` to events as nullable TIMESTAMPTZ with no default
**When to use:** Whenever adding a flag/timestamp column that should not affect existing rows
**Example:**
```typescript
// src/lib/db/schema.ts — add to events column definitions
archived_at: timestamp('archived_at', { withTimezone: true }), // nullable, no .notNull()
```
Generated SQL (drizzle-kit produces):
```sql
ALTER TABLE "events" ADD COLUMN "archived_at" timestamp with time zone;
```
- No default clause = existing rows get NULL automatically
- No `.notNull()` = column is nullable in Drizzle type (`Date | null`)
- `InferSelectModel<typeof events>.archived_at` becomes `Date | null` automatically

### Pattern 2: New Table with FK and Unique Index
**What:** Create `recurring_series` table and add `series_id` FK to events
**When to use:** When introducing a new entity with a relationship to an existing table
**Example:**
```typescript
// src/lib/db/schema.ts — new table
export const recurring_series = pgTable(
  'recurring_series',
  {
    id: serial('id').primaryKey(),
    venue_id: integer('venue_id')
      .references(() => venues.id)
      .notNull(),
    normalized_performer: text('normalized_performer').notNull(),
    created_at: timestamp('created_at').defaultNow().notNull(),
    updated_at: timestamp('updated_at').defaultNow().notNull(),
  },
  (table) => [
    uniqueIndex('recurring_series_venue_performer_key').on(
      table.venue_id,
      table.normalized_performer
    ),
  ]
);

// In events table columns, add:
series_id: integer('series_id')
  .references(() => recurring_series.id), // nullable, no .notNull()
```
Generated SQL (drizzle-kit produces):
```sql
CREATE TABLE "recurring_series" (
  "id" serial PRIMARY KEY NOT NULL,
  "venue_id" integer NOT NULL,
  "normalized_performer" text NOT NULL,
  "created_at" timestamp DEFAULT now() NOT NULL,
  "updated_at" timestamp DEFAULT now() NOT NULL
);
ALTER TABLE "recurring_series" ADD CONSTRAINT "recurring_series_venue_id_venues_id_fk"
  FOREIGN KEY ("venue_id") REFERENCES "public"."venues"("id") ON DELETE no action ON UPDATE no action;
ALTER TABLE "events" ADD COLUMN "series_id" integer;
ALTER TABLE "events" ADD CONSTRAINT "events_series_id_recurring_series_id_fk"
  FOREIGN KEY ("series_id") REFERENCES "public"."recurring_series"("id") ON DELETE no action ON UPDATE no action;
CREATE UNIQUE INDEX "recurring_series_venue_performer_key"
  ON "recurring_series" USING btree ("venue_id","normalized_performer");
```

### Pattern 4: Migration Workflow
```bash
# 1. Edit src/lib/db/schema.ts
# 2. Generate migration SQL
npm run db:generate
# drizzle-kit diffs against 0010_* snapshot → writes 0011_*.sql

# 3. Apply to Neon Postgres
npm run db:migrate
# drizzle-kit executes pending SQL against DATABASE_URL

# 4. Verify schema.test.ts passes
npx jest src/lib/db/schema.test.ts
```

### Anti-Patterns to Avoid
- **Applying raw SQL to Neon console without running db:generate:** Puts Drizzle snapshot out of sync — `db:generate` will produce wrong diff on next run.
- **Setting `series_id` as `.notNull()`:** Events have no series at insert time; Phase 31 populates it in a backfill pass.
- **Giving `archived_at` a default value:** `DEFAULT NULL` is implicit when no `.default()` is called; making it `.defaultNow()` would wrongly archive all existing rows during migration.
- **Putting `recurring_series` import into client.ts schema object without exporting it from schema.ts first:** The `drizzle({ client: sql, schema })` import in client.ts pulls `* as schema` — the new table is automatically included once exported from schema.ts.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Migration SQL | Hand-write ALTER TABLE scripts | `npm run db:generate` | drizzle-kit tracks snapshot state; manual SQL bypasses it and causes drift |
| TypeScript type for new columns | Manual `Event & { archived_at: Date or null }` type extension | `InferSelectModel<typeof events>` in types/index.ts | Type propagates automatically; manual extension creates duplicate/divergent types |
| Unique constraint for venue-scoped series | Application-level dedup check before insert | `uniqueIndex('...').on(table.venue_id, table.normalized_performer)` | DB-level constraint is atomic and prevents race conditions |
| Nullable column with NULL default | `DEFAULT NULL` explicit in SQL | Drizzle omitting `.notNull()` and `.default()` | Standard Drizzle pattern; DB default for nullable column is already NULL |

**Key insight:** In this codebase all schema changes flow through `schema.ts` → `db:generate` → `db:migrate`. The snapshot in `drizzle/meta/` is the source of truth for drizzle-kit's diff algorithm. Skipping any step breaks subsequent migrations.

## Common Pitfalls

### Pitfall 1: event_date is TIMESTAMP (no TZ), archived_at should be TIMESTAMPTZ
**What goes wrong:** Using plain `timestamp()` for `archived_at` when the spec requires TIMESTAMPTZ, or attempting to "fix" `event_date` in the same migration.
**Why it happens:** Drizzle's `timestamp()` without `{ withTimezone: true }` maps to `TIMESTAMP` (no zone). The existing `event_date` column was created this way and is unchanged.
**How to avoid:** Use `timestamp('archived_at', { withTimezone: true })` for the new column only. Do not touch `event_date` — it is out of scope and changing its type would be a destructive migration requiring careful analysis.
**Warning signs:** If the generated SQL says `timestamp with time zone` for `archived_at`, that is correct. If it says just `timestamp`, the option was omitted.

### Pitfall 2: Forward-Reference in series_id FK Definition
**What goes wrong:** Drizzle evaluates table definitions sequentially; `events.series_id` referencing `recurring_series.id` requires `recurring_series` to be declared first in the file, or using a callback pattern.
**Why it happens:** JavaScript module evaluation order — the referenced table must be initialized before it is referenced.
**How to avoid:** Define `recurring_series` before `events` in schema.ts, OR use an arrow function callback: `.references(() => recurring_series.id)`. The existing `events.venue_id` uses the callback form (`.references(() => venues.id)`), which defers evaluation — use the same pattern.
**Warning signs:** TypeScript `Cannot access 'recurring_series' before initialization` at module load.

### Pitfall 3: drizzle-kit Snapshot Drift
**What goes wrong:** Making changes to schema.ts, running the app but forgetting to run `db:generate` + `db:migrate`, then later running `db:generate` which produces a migration that overwrites/conflicts with manual SQL already in the DB.
**Why it happens:** drizzle-kit compares schema.ts against the last snapshot in `drizzle/meta/`, not against the live DB.
**How to avoid:** Always run `db:generate` → `db:migrate` as a pair immediately after editing schema.ts. Never apply schema changes to the live DB outside this workflow.

### Pitfall 4: schema.test.ts Must Be Updated
**What goes wrong:** Adding columns to schema.ts but not the corresponding column presence assertions in schema.test.ts causes the test to pass but silently miss the new fields.
**Why it happens:** schema.test.ts validates specific expected column lists — it won't fail on new additions, but it also won't confirm they exist.
**How to avoid:** Add `'archived_at'`, `'series_id'` to the events expected columns array, and add a new `recurring_series` describe block testing its column structure.

### Pitfall 5: Zod v4 `.nullable()` vs `.optional()` Distinction
**What goes wrong:** Using `.nullable()` instead of `.optional()` for `recurrence_pattern`.
**Why it happens:** `.nullable()` allows `null` explicitly; `.optional()` allows the key to be absent. For an AI extraction hint, the field simply won't be present in most responses — `.optional()` is correct.
**How to avoid:** Use `z.string().optional()` — field may be absent from the response object entirely. If you also want to allow explicit `null`, use `z.string().nullish()`.

## Open Questions

1. **Should recurring_series include additional metadata columns now or defer to Phase 31?**
   - What we know: Phase 31 (detection) populates the table; Phase 29 only needs the structural columns that Phase 30's COALESCE guard references indirectly
   - What's unclear: Whether SER-02 detection needs columns beyond `(id, venue_id, normalized_performer)` — e.g., `weekday`, `interval_days`, `confidence`
   - Recommendation: Define only `(id, venue_id, normalized_performer, created_at, updated_at)` in Phase 29. Phase 31 can add detection-specific columns in a new migration. This keeps Phase 29 minimal and non-breaking.

2. **event_date stored as plain TIMESTAMP — impact on Phase 30 archival**
   - What we know: `event_date timestamp NOT NULL` — no timezone. Confirmed in drizzle/0000 migration and snapshot.
   - What's unclear: Whether existing data was inserted as UTC or as Atlantic local time
   - Recommendation: Flag for Phase 30. The archival cron (Phase 30) must explicitly apply Atlantic offset (`America/Halifax`, UTC-4 in summer / UTC-3:30 in NL). Not a Phase 29 concern but must be documented in the handoff.
