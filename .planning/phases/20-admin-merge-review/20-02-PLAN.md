  <action>
    **page.tsx — Server component:**

    Follow the discovery page pattern exactly (`export const dynamic = 'force-dynamic'`).

    1. Parse `searchParams: Promise<{ status?: string }>` (Next.js 15 async params)
    2. Validate status against `['pending', 'merged', 'kept_separate']`, default to `'pending'`
    3. Use Drizzle `alias(venues, 'venue_a')` and `alias(venues, 'venue_b')` for self-join
    4. Run parallel queries:
       - Main query: select candidate fields + venue_a fields + venue_b fields, inner join on both venue FKs, filter by status, order by created_at desc
       - Count queries: pending count, merged count, kept_separate count
    5. For each candidate row, also need event count and source count per venue. Since the queue is expected to be small, collect all unique venue IDs from the result set and run two GROUP BY queries:
       - `SELECT venue_id, COUNT(*) FROM events WHERE venue_id IN (...) GROUP BY venue_id`
       - `SELECT venue_id, COUNT(*) FROM scrape_sources WHERE venue_id IN (...) GROUP BY venue_id`
       Then build lookup maps and attach counts to each candidate before passing to the client component.
    6. Render page header: "Merge Review" title with subtitle "Review and resolve borderline venue merge candidates."
    7. Render `<MergeReviewList>` passing candidates (with counts attached), tab counts, and active status

    **MergeReviewList.tsx — Client component ('use client'):**

    Follow the DiscoveryList pattern for tabs. Import `mergePair` and `keepSeparate` from `../actions`.

    1. **Tab bar:** Three tabs (Pending, Merged, Kept Separate) with counts in parentheses. Each tab is a link to `?status=pending|merged|kept_separate`. Active tab styled bold. Match discovery page tab styling.

    2. **Empty state:** When no candidates: "No pending merge candidates. Candidates appear after Ticketmaster ingests detect near-match venues."

    3. **Candidate cards:** For each candidate, render a side-by-side layout:
       - Left card: Venue A (name, city, province, lat/lng if available, event count, source count)
       - Center metadata: name_score (formatted as percentage, e.g. "92% name match"), distance_meters (formatted as "150m apart" or "No coordinates"), reason as human-readable label
       - Right card: Venue B (same fields as left)

       Reason labels map:
       ```
       name_match_no_geo: "Name match, no coordinates"
       name_match_geo_distant: "Name match, locations too far apart"
       name_match_geo_uncertain: "Name match, locations borderline distance"
       geo_close_name_differs: "Locations close, names differ"
       ```

    4. **Action buttons** (below card pair, only for pending tab):
       - "Merge" button (blue): Uses inline confirmation pattern.
         - First click: sets `confirmingId` state to this candidate's id
         - Shows "Confirm merge?" button (red) + "Cancel" button
         - "Confirm merge?" submits form with `action={mergePair}` and hidden input `candidateId`
         - "Cancel" resets `confirmingId` to null
         - Use `useFormStatus` for pending state on submit button (disable + show "Merging...")
       - "Keep Separate" button (gray): Single form submit with `action={keepSeparate}` and hidden input `candidateId`. No confirmation needed per user decision.

    5. **For merged/kept_separate tabs:** Show the same card layout but without action buttons. Optionally show `reviewed_at` timestamp (Claude's discretion — show it, formatted as relative time or date string).

    Styling: Use Tailwind consistent with existing admin pages. Cards with `bg-white rounded-lg shadow-sm border p-4`. Responsive: stack vertically on mobile (flex-col), side-by-side on md+ (flex-row).
