## Summary

Phase 15 is a pure instrumentation and display phase. It has no new dependencies, no new routes, and no new modules. All building blocks are already in place from Phase 14: the schema already has `max_pages` added via Drizzle migration, `orchestrator.ts` already writes `last_scraped_at` and `last_scrape_status` on success and failure paths, and `admin/page.tsx` already renders a source health table.

The work is: (1) add five metric columns to `scrape_sources` in `schema.ts`, (2) generate and run the Drizzle migration, (3) extend the two DB update calls in `orchestrator.ts` to also write the metric values, and (4) extend the admin dashboard page query and table to display the three new columns required by SCRP-04 (`last_event_count`, `avg_confidence`, `consecutive_failures`), with a visual flag when `consecutive_failures >= 3`.

**Primary recommendation:** Instrument the orchestrator first, verify values appear in the DB, then extend the UI. The schema migration is the hard prerequisite — nothing else is unblockable once columns exist.

### Pattern 1: Additive Schema Column Addition

**What:** Add new nullable or defaulted columns to an existing table, generate one migration, run it. No existing rows or queries break.

**When to use:** Always — destructive schema changes require downtime coordination; additive changes are safe on a live production app.

**Example (from ARCHITECTURE.md — HIGH confidence, matches existing schema.ts style):**

```typescript
// src/lib/db/schema.ts — additions to scrape_sources table definition
export const scrape_sources = pgTable('scrape_sources', {
  // ... existing columns unchanged ...
  max_pages: integer('max_pages').notNull().default(1), // already added in Phase 14

  // Phase 15 additions:
  last_event_count:       integer('last_event_count'),                              // nullable — null means never scraped
  avg_confidence:         doublePrecision('avg_confidence'),                        // nullable — null means never scraped
  consecutive_failures:   integer('consecutive_failures').notNull().default(0),     // tracks consecutive failures; resets on success
  total_scrapes:          integer('total_scrapes').notNull().default(0),            // lifetime total scrape attempts
  total_events_extracted: integer('total_events_extracted').notNull().default(0),   // lifetime events extracted
});
```

**Migration generation command:**
```bash
npx drizzle-kit generate
```

**Migration run command:**
```bash
npx drizzle-kit migrate
```

**Critical: Export pgEnum definitions** — confirmed open Drizzle bug #5174: `pgEnum` values are silently omitted from migration SQL if not exported at module level. `eventCategoryEnum` is already exported in schema.ts — ensure it remains exported after any edits.

### Pattern 2: SQL-Level Increment for Counters

**What:** Use Drizzle's `sql` template tag for atomic counter increments rather than fetch-increment-write (which is not safe for concurrent processes and requires two DB round-trips).

**When to use:** Any monotonically-increasing column (`total_scrapes`, `total_events_extracted`, `consecutive_failures`).

**Example (from ARCHITECTURE.md):**

```typescript
import { sql, eq } from 'drizzle-orm';

// On success:
await db.update(scrape_sources).set({
  last_scraped_at:        new Date(),
  last_scrape_status:     'success',
  last_event_count:       eventCount,
  avg_confidence:         avgConf,             // null when eventCount === 0
  consecutive_failures:   0,                   // reset on success
  total_scrapes:          sql`total_scrapes + 1`,
  total_events_extracted: sql`total_events_extracted + ${eventCount}`,
}).where(eq(scrape_sources.id, source.id));

// On failure:
await db.update(scrape_sources).set({
  last_scraped_at:      new Date(),
  last_scrape_status:   'failure',
  consecutive_failures: sql`consecutive_failures + 1`,
  total_scrapes:        sql`total_scrapes + 1`,
  // last_event_count and avg_confidence are NOT touched on failure — preserve last known values
}).where(eq(scrape_sources.id, source.id));
```

**Where these writes go in orchestrator.ts:** The existing success write is at line 108–111 and the failure write is at line 113–118. Both are already inside the correct `try/catch` structure. Phase 15 extends these two `db.update()` calls — no structural change to the loop.

### Pattern 3: Average Confidence Calculation

**What:** Compute the mean confidence of extracted events in the orchestrator immediately after extraction, before the upsert loop. Store as `avg_confidence` (nullable `doublePrecision`).

**When to use:** Any per-source run where events were extracted.

**Example:**

```typescript
// Compute after extraction, before upsert loop
const eventCount = extracted.length;
const avgConf = eventCount > 0
  ? extracted.reduce((sum, e) => sum + (e.confidence ?? 0), 0) / eventCount
  : null;   // null when no events found — distinguishes "zero events" from "never scraped"
```

**Note on eventbrite and bandsintown sources:** These handlers (`scrapeEventbrite`, `scrapeBandsintown`) do not currently return `ExtractedEvent[]` back to the orchestrator — they call `upsertEvent` internally. For Phase 15, the simplest approach is to write metric values for these source types with `last_event_count: null` and `avg_confidence: null` (just update the counters). Alternatively, refactor handlers to return counts — but that creates scope creep. Keep Phase 15 scope minimal: instrument the `venue_website` branch fully, and write minimal counter updates for other source types.

### Pattern 4: Visual Failure Flag in Admin Table

**What:** Inline conditional styling or a badge component on the `consecutive_failures` cell. Follows the existing `statusBadge()` pattern in `admin/page.tsx`.

**When to use:** When `consecutive_failures >= 3`, display a warning indicator (amber/red background) to prompt admin attention.

**Example (consistent with existing badge style in admin/page.tsx):**

```typescript
function failuresBadge(count: number | null) {
  if (count == null) return <span className="text-gray-400 text-sm">—</span>;
  if (count >= 3) {
    return (
      <span className="inline-flex items-center px-2.5 py-0.5 rounded-full text-xs font-medium bg-amber-100 text-amber-800">
        {count} failures
      </span>
    );
  }
  return <span className="text-sm text-gray-700">{count}</span>;
}
```

**Threshold:** 3 consecutive failures is the specified threshold per the success criteria. This is not configurable — hard-coded in the UI conditional.

**Current query (lines 83–96 of admin/page.tsx):**

The `sourceRows` array type and the JSX table must both be updated to include the three new fields.

### Anti-Patterns to Avoid

- **Touching `last_event_count` on failure path:** Preserve last known event count on failure — do not overwrite with 0 or null. This preserves the last good scrape result for the admin.
- **Calling `scrapeEventbrite`/`scrapeBandsintown` and computing metrics from their return value:** These handlers return `void`. Do not restructure them in Phase 15 — write a minimal counter update for non-venue-website sources.
- **Bypassing the metrics write for any source type:** The success and failure writes must run for ALL source types in the loop, including `eventbrite` and `bandsintown`. The existing `try/catch` structure handles this — extend both the success path (line 108) and the failure path (line 113) in orchestrator.ts.
- **Merging metric writes into the handler modules:** Metric writes belong in `orchestrator.ts`, not inside `eventbrite.ts` or `bandsintown.ts`. The orchestrator owns the per-source lifecycle.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Atomic counter increments | Fetch current value + increment + write | `sql\`column + 1\`` in Drizzle update | Race condition safe; one DB round-trip |
| Average calculation | Store all confidence values in a separate table | `extracted.reduce()` in orchestrator | Simple per-run mean is sufficient for health monitoring; historical trend is out of scope |
| Alert/notification on 3+ failures | Email, webhook, or background job | Visual flag in admin table | Out of scope per requirements; real-time monitoring explicitly out of scope (REQUIREMENTS.md) |

## Common Pitfalls

### Pitfall 1: Drizzle pgEnum Export Bug

**What goes wrong:** Drizzle migration silently omits `pgEnum` values from the generated SQL if they are not explicitly exported at module level.

**Why it happens:** Open bug #5174 in Drizzle ORM. The migration generator inspects only exported symbols.

**How to avoid:** `eventCategoryEnum` is already exported in `schema.ts`. After editing, verify it remains `export const eventCategoryEnum = ...`. Do not remove the export.

**Warning signs:** Generated migration SQL contains `ALTER TABLE` statements but no `CREATE TYPE` — check the SQL file immediately after `drizzle-kit generate`.

### Pitfall 2: Overwriting Last Known Event Count on Failure

**What goes wrong:** Failure path sets `last_event_count: 0`, losing the last known good event count. Admin can no longer distinguish "was scraping fine, now failing" from "never returned events".

**Why it happens:** Treating failure path symmetrically with success path.

**How to avoid:** On failure, do NOT include `last_event_count` or `avg_confidence` in the `db.update().set()` call. Only update `last_scraped_at`, `last_scrape_status`, `consecutive_failures`, and `total_scrapes`.

### Pitfall 3: Metric Writes Only on venue_website Branch

**What goes wrong:** Developer adds metric writes inside the `if (source.source_type === 'venue_website')` block. Eventbrite and Bandsintown sources never get their counters updated. SCRP-04 requires per-source metrics for "each source".

**Why it happens:** The `last_event_count` and `avg_confidence` computation logically lives inside the `venue_website` branch where `extracted` is available. The counter writes get placed there too.

**How to avoid:** The existing success write (`last_scraped_at`, `last_scrape_status`) already sits OUTSIDE and AFTER the source_type dispatch. Add the counter increments to that same write. Compute `last_event_count` and `avgConf` inside the `venue_website` branch, but write all of them in the single `db.update()` outside the branch. For `eventbrite`/`bandsintown`, use `null` for `last_event_count` and `avg_confidence` (or track them separately if needed in a later phase).

### Pitfall 4: TypeScript Error — `sql` Template Tag Misuse

**What goes wrong:** `sql\`column + 1\`` fails TypeScript typecheck because the Drizzle `set()` method expects typed values, not raw SQL fragments, for columns typed as `integer`.

**Why it happens:** Drizzle's `set()` method signature accepts `SQL` objects for columns, but TypeScript inference may complain.

**How to avoid:** Import `sql` from `drizzle-orm` (not from the db client). The `sql` template tag returns a `SQL` object that Drizzle accepts in `set()`. Pattern used in existing codebase — verified that `import { sql, eq } from 'drizzle-orm'` is the correct import path.

### Pitfall 5: Admin Table Row Type Not Updated

**What goes wrong:** The `sourceRows` array is typed inline (lines 53–60 in admin/page.tsx). After extending the query, the TypeScript type for `sourceRows` must be updated to include the three new fields. If not updated, the JSX that references `row.consecutiveFailures` will have a TypeScript error.

**How to avoid:** Let TypeScript infer the type from the query result. Remove the explicit `Array<{ ... }>` type annotation on `sourceRows` and let `typeof sourceHealthResult` drive the type, or update the inline type to include the new fields.

## Open Questions

1. **Metric writes for eventbrite/bandsintown source types**
   - What we know: These handlers return `void`; `extracted` is not available in the orchestrator loop for these types. The `total_scrapes` counter can still be incremented.
   - What's unclear: Whether admins want `last_event_count` for API-based sources too. API sources typically succeed or fail cleanly; event counts are less useful than for scraped HTML.
   - Recommendation: Write `null` for `last_event_count` and `avg_confidence` for non-venue-website sources in Phase 15. If this matters later (Phase 16 Ticketmaster adds a third API source), revisit then. The counter columns (`total_scrapes`, `consecutive_failures`) will still work for all source types.

2. **avg_confidence display precision**
   - What we know: `doublePrecision` stores full floating point. Displaying `0.8666666666...` in the admin table would be ugly.
   - What's unclear: Whether to round in the SQL query, in the JSX, or not at all.
   - Recommendation: Round to 2 decimal places in JSX (`(row.avgConfidence?.toFixed(2) ?? '—')`). No SQL-level rounding needed — keep raw precision in the DB for future analytics.
